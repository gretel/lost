# Multi-CR Support — Implementation Plan

## Overview

MeshCore repeaters are switching from CR8 (4/8, internal cr=4) to CR5 (4/5, internal cr=1) for spectral efficiency. The LoRa PHY core algorithms (hamming encode/decode, interleaving, DecodeChain) already support CR 1-4, but three gaps prevent real-world CR5 operation:

1. **Bridge CR passthrough broken** — bridge stores CR inconsistently (1-4 or 5-8 depending on code path) and never passes CR to lora_trx in CBOR TX requests
2. **CLI `lora tx` missing `--cr`** — no way to override TX CR from command line
3. **CR=3 untested** — test vectors and multi-config matrices skip CR=3 (CR=2 already implicitly works)

## Desired End State

- Repeater bridge can be configured to any CR (5-8) via companion `SET_RADIO_PARAMS`
- Bridge correctly passes CR to lora_trx so TX uses the configured rate
- `lora tx` CLI supports `--cr` override
- CR=3 has test vector coverage and passes end-to-end

## What We're NOT Doing

- No parallel multi-CR decoder lanes (CR is per-frame metadata from explicit header, already works)
- No changes to the C++ core PHY algorithms (they handle CR 1-4 correctly)
- No soft decoder LUT rewrite (the shared-LUT shift works mathematically; Phase 4 validates it)
- No separate decode-only chain for different CR values

## Phase 1: Fix Bridge CR Convention + Passthrough

### Overview

Bridge `state.cr` currently stores CR inconsistently: internal 1-4 from `lora_trx` config path, external 5-8 from `SET_RADIO_PARAMS`. Outgoing `lora_tx` CBOR messages never include `cr`, so TX always uses lora_trx's TOML-configured CR regardless of companion setting.

Fix: normalize `state.cr` to external 5-8 (companion protocol convention), add CR to all bridge TX CBOR.

### Changes Required:

#### 1a. Fix CR init in `cli.py`

**File**: `scripts/src/lora/bridges/meshcore/cli.py`

**Changes**:

- Line ~404: Change `cr = int(phy.get("cr", 8)) - 4` to `cr = int(phy.get("cr", 4)) + 4`
  - Reason: lora_trx sends CR as internal (1-4) in config CBOR. Bridge needs to store external (5-8).
  - Default changes from 8 (external) to 4 (internal) to match lora_trx's encoding.
  - Then +4 converts 1-4 → 5-8.

#### 1b. Fix CR config handler in `run_bridge()` in `cli.py`

**File**: `scripts/src/lora/bridges/meshcore/cli.py`

**Changes**:

- Line ~203: The `config` message handler currently does `state.cr = int(phy.get("cr", state.cr + 4)) - 4`.
  - Change to `state.cr = int(phy.get("cr", state.cr - 4)) + 4`
  - lora_trx sends internal 1-4; bridge stores external 5-8.
  - Fallback: if phy.cr is absent, derive from existing state.cr by converting back (`state.cr - 4`).

#### 1c. Add CR to outgoing bridge TX CBOR

**File**: `scripts/src/lora/tools/meshcore_tx.py`

**Changes**:

- `make_cbor_tx_request(packet, **phy_overrides)` — add a `cr` keyword or have it always include CR
  - Simplest: add `cr` parameter to the function signature. Callers pass state.cr (external 5-8), function converts to internal (`cr - 4`) before encoding.
  - Or: change function signature to `make_cbor_tx_request(packet, cr=0, **phy_overrides)` where 0 means "don't include".

**File**: `scripts/src/lora/bridges/meshcore/cli.py`

**Changes**:

- Every call to `make_cbor_tx_request()` needs to pass `state.cr`:
  - Line ~585: ACK TX: `make_cbor_tx_request(ack_pkt, cr=state.cr)`
  - Line ~592: repeat forward: `make_cbor_tx_request(fwd_pkt, cr=state.cr)`

**File**: `scripts/src/lora/bridges/meshcore/companion.py`

**Changes**:

- All calls to `make_cbor_tx_request()` need to pass `state.cr`:
  - `_handle_send_txt_msg()` — has access to `state`
  - `_handle_send_advert()` — has access to `state`
  - `_handle_send_chan_txt_msg()` — has access to `state`
  - `_handle_send_control_data()` — has access to `state`
  - `_handle_anon_req_rx()` — has `state` in scope
  - etc.

#### 1d. Normalize SET_RADIO_PARAMS CR handling

**File**: `scripts/src/lora/bridges/meshcore/companion.py`

**Changes**:

- Line ~1306: `cr_new = data[9]; state.cr = cr_new`
  - Already external (5-8) from companion protocol. Add validation: clamp to [5, 8].
  - This is already correct but add explicit range check.

## Phase 4: Validate Soft Decoder for CR=2,3

### Overview

The soft ML decoder uses `hamming_cw_LUT` (8-bit codeword LUT) for all CR≥2, right-shifted by `(8 - cw_len)` to align with the actual codeword width. This works mathematically but has no explicit test coverage for CR=2 (6-bit) or CR=3 (7-bit). Add targeted tests.

### Changes Required:

#### 4a. Soft decoder tests for all CR values

**File**: `test/qa_lora_hamming.cpp` (or similar existing test file)

**Changes**:

- Add explicit `hamming_decode_soft()` test cases for CR_app=1,2,3,4.
- For each CR_app: encode all 16 nibbles with `hamming_encode()`, corrupt one or more bits with varying LLR signs, verify `hamming_decode_soft()` recovers the correct nibble.
- Edge case: CR_app=2 should detect but never correct (syndrome-based correction is skipped; verify soft ML still finds correct codeword via LUT).
- Edge case: CR_app=3 should correct single bit errors (verify).

### Success Criteria:

#### Automated Verification:

- [ ] All 16 nibbles at CR=4 decode correctly under soft ML with no noise
- [ ] All 16 nibbles at CR=4 decode correctly under soft ML with single-bit inversion
- [ ] All 16 nibbles at CR=3 decode correctly under soft ML with single-bit inversion
- [ ] All 16 nibbles at CR=2 decode correctly under soft ML with single-bit inversion
- [ ] All 16 nibbles at CR=1 decode correctly under soft ML with single-bit inversion
- [ ] `ctest -R qa_lora_hamming -j$(nproc)` passes

## Testing Strategy

### Automated:

- Full test suite: `ctest --test-dir build -j$(nproc) -R qa_lora`
- All unit tests in `test/qa_lora_*.cpp`
- Existing tests must not regress

### Manual Testing Steps:

1. Start lora_trx with CR=5 in config: `cr = 5` in TOML → verify TX uses 4/5
2. Start bridge connected to lora_trx → verify companion app shows CR 4/5
3. Send CR=8 frame via `lora tx --cr 8` → verify lora_trx TXes at CR8
4. Verify reception at different CR: TX CR=5 frame → RX DecodeChain reads header CR correctly → FrameResult.cr == 1

## Performance Considerations

- CR=5 (4/5) has 1 parity bit per 4 data bits → 20% overhead instead of 100% for CR=8 (4/8)
- Shorter airtime per frame → higher throughput, lower collision probability
- CR=5 has zero error correction (single parity detection only) → weaker in noisy channels
- No CPU impact from CR change (interleaver/Hamming are lightweight)
