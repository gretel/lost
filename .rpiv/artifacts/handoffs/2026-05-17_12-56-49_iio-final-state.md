# Handoff: IIO header decode — root cause found, AA filter non-viable, stride path detects but can't decode

## Task(s)

**Completed:**

1. ✅ **Root cause confirmed** — spectral aliasing from stride decimation at OS=40. Post-decimation Nyquist = 31.25 kHz = LoRa BW/2. Rectangular chirp envelope spectral leakage aliases back into band → corrupts PreambleSync FFT peaks → wrong bin_hat (varies 3-255, should cluster near 0-7).

2. ✅ **Decode chain verified correct** — deinterleaver, Hamming(8,4), checksum all work. 2 coincidental csum=1 frames across ~15 test runs (both with wrong CR values — statistical noise, not real decodes).

3. ✅ **AA filter at OS=64 ruled out** — 6-stage half-band cascade reduces PMR 3.2× (256→81 for SF8 in perfect conditions). With weak Heltec V3 companion (2 dBm), SF8 detection drops to 0-1 per test. SF7 occasionally detects (3-5 per test). Gain increase from 60→71 dB doesn't help — PMR loss is fundamental, not SNR-limited.

4. ✅ **CFO strategy**: CFO=0 best for stride path (1 coincidental csum=1). PreambleSync CFO estimates corrupted by aliasing (cfo_int range -63 to +55 at OS=40, expected ~7). Using PreambleSync estimates makes decode worse.

5. ✅ **decim_phase fix** correct but insufficient — tau_late handles integer STO, decim_phase from sto_frac handles fractional.

6. ✅ **12-bin filter delay compensation** for AA path — Python simulation confirmed 6-stage half-band cascade shifts dechirped peak by exactly 12 bins (all SF values). Added to CFO correction when `aa_enabled=true`.

7. ✅ **Vangelista threshold disabled** (`verify_threshold = false`) — helps AA filter detect more SF7 frames but doesn't rescue SF8.

8. ✅ **TX loopback broken** — `lora tx advert --connect 10.0.23.149:5561` sends CBOR, UDP client subscribes, but IIOSink never transmits (no `tx_ack`, no TX cbor in logs). All frames during TX loopback tests were false noise locks.

## Learnings

1. **Root cause is aliasing, not CFO/timing.** At OS=40, Nyquist after stride = sample_rate/(2×os) = BW/2 = 31.25 kHz. LoRa BW = 62.5 kHz spans exactly [-31.25, +31.25] kHz. The rectangular chirp envelope has spectral leakage beyond the Nyquist boundary → aliases back into band → creates spurious FFT peaks that move randomly (bin_hat varies 3-255). This corrupts PreambleSync detection, CFO estimation, and STO estimation.

2. **AA filter halves problem — solves aliasing but introduces PMR loss.** The 6-stage half-band cascade at OS=64 produces correct sample counts and magnitudes (verified by Python simulation and AA_STAGE diag). But cumulative filter response at band edge attenuates chirp frequencies near ±31.25 kHz → amplitude-modulates the chirp → broadens dechirped peak → PMR drops 3.2×. For weak companion signal (2 dBm Heltec V3), PMR drops below detection threshold for SF8. Gain doesn't fix this — it's signal self-distortion, not SNR.

3. **Half-band filter has linear phase — group delay is NOT the issue.** Python simulation confirmed symmetric FIR coefficients → linear phase → constant group delay. The 12-bin shift is from group delay (deterministic, compensable). The real issue is passband non-flatness at band edge causing chirp amplitude modulation.

4. **CFO=0 better than PreambleSync estimates at OS=40.** PreambleSync CFO estimates are corrupted by aliasing (cfo_int range -63 to +55, expected ~7 from 2ppm TCXO). Using these estimates makes decode WORSE than using zero. The actual CFO on this PlutoSDR appears near zero (AD9361 calibration likely active).

5. **Decode chain works.** 2 coincidental csum=1 across ~15 tests prove deinterleaver, Hamming(8,4), and checksum all correct. The failure is upstream — demodulated bins are wrong due to aliasing.

6. **TX loopback is broken — don't use it.** `lora tx advert` sends CBOR, UDP client connects, but IIOSink never fires. No `tx_ack` in logs. Companion (10.0.23.152:5000) is the only working signal source.

7. **CI takes ~25-30 min** for armv7 cross-build. Trigger: `gh workflow run ci-armv7.yml --repo gretel/chirpmunk-gr4 --ref iio-on-device`.

## Action Items & Next Steps

### Primary: Fix stride aliasing

The OS=40 stride path is the only one that reliably detects SF8 frames (7-10 per test). But bins are always wrong due to aliasing. Options:

1. **Linear-phase FIR pre-filter** — replace half-band cascade with a proper linear-phase FIR lowpass filter that supports arbitrary decimation ratios (not just powers of 2). More taps, better passband flatness. Key: must preserve chirp constant-envelope property.

2. **Change sample rate to get power-of-2 OS** — AD9361 valid rates: 40MHz/N. OS=32 needs 2.0 MS/s (N=20). Driver rejected 2.0 MS/s previously (`EINVAL` from `iio_channel_attr_write_longlong`). May be fixable via different IIO API or firmware update.

3. **Increase companion TX power** — Heltec V3 SX1262 max +22 dBm with PA boost. Currently at 2 dBm. Meshcore firmware may support `CMD_SET_RADIO` with power parameter, or may need firmware modification.

### Secondary: Make TX loopback work

4. **Debug IIOSink TX path** — `lora tx advert` CBOR is received (UDP client subscribes) but IIOSink never fires. Check `apps/tx_worker.cpp` TX CBOR handling, IIOSink buffer management, and `set_mode_at_init` interactions.
