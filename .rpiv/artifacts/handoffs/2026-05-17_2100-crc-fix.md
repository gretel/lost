# Handoff: Payload CRC-16 — 3 bugs fixed

## Task(s)

### [COMPLETED] Task 1: Fix payload CRC algorithm (XOR-2 → plain CRC-16/XMODEM)

**Root cause**: `lora_payload_crc` (used by TX `add_crc` and RX `lora_verify_crc`) used a non-standard XOR-2 variant incompatible with SX1262.

```
OLD: CRC16(payload[0..N-3]) ^ payload[N-2] ^ (payload[N-1] << 8)
NEW: CRC16(payload[0..N-1])  // poly 0x1021, init 0x0000
```

**Fix** (commit `58b9765`): Both `lora_payload_crc` and `lora_verify_crc` now use plain `crc16()`.

### [COMPLETED] Task 2: Fix \_frame_symb_numb for implicit mode

**Root cause**: DecodeChain::reset() unconditionally added 8u + to frame length for implicit mode. Implicit frames have no header symbols — frame length should be `computePayloadSymbols()` only. The +8 caused the decoder to read 8 symbols into the next frame before finishing, corrupting CRC.

**Fix** (commit `0c0d784`): Removed `8u +` from implicit mode path.

### [COMPLETED] Task 3: Fix CRC dewhitening

**Root cause**: SX1262 whitens the entire data stream including CRC nibbles. The `dewhiten()` function only dewhitens the first `pay_len` bytes. CRC bytes at `bytes[pay_len]` and `bytes[pay_len+1]` were left in whitened form, so the CRC comparison was against whitened CRC bytes vs unwhitened computed CRC.

**Fix** (commit `0c0d784`): After `dewhiten()`, XOR CRC bytes with `whitening_seq[pay_len]` and `whitening_seq[pay_len+1]` to recover original CRC. Since nibble-level dewhitening is equivalent to full-byte XOR (nibbles occupy non-overlapping bit positions), `bytes[k] ^= whitening_seq[k]` recovers the dewhitened byte.

### [PENDING] Task 4: TX CRC whitening (for SX1262 compatibility)

The `add_crc` function in `tx_chain.hpp` appends raw CRC nibbles after whitened payload. For TX to SX1262 receivers, CRC nibbles should also be whitened. Currently not urgent — only RX path is active.

CI run 25998349777 succeeded (4m41s). Binary deployed to tezuka, tested with companion ADVERT.

## Learnings

1. **SX1262 CRC is plain CRC-16/XMODEM** over all payload bytes. No XOR-2 variant.

2. **SX1262 whitens CRC bytes** as part of the data stream. All data nibbles after the header (payload + CRC) are XOR'd with the whitening sequence.

3. **Nibble-level dewhitening is equivalent to byte-level XOR**. Since `dewhiten()` assembles `byte = (hi << 4) | lo` and the whitening nibbles (ws_lo, ws_hi) occupy non-overlapping bit positions in the byte, `byte ^ ws` accomplishes the same as nibble-by-nibble XOR. This makes post-hoc CRC dewhitening trivial (`bytes[pay_len] ^= whitening_seq[pay_len]`).

4. **Implicit mode has no header symbols**. `_frame_symb_numb` must NOT include the +8 header offset that explicit mode uses. Without this fix, the decoder consumes 8 symbols from the next frame.

5. **Three independent bugs masked each other**:
   - Wrong CRC polynomial algorithm
   - Missing CRC dewhitening
   - Wrong frame length for implicit mode
     Fixing any single one was insufficient; all three had to align.

## Other Notes

- **CRC dewhitening insight**: The equivalence of nibble-level and byte-level XOR
  holds because nibbles occupy disjoint bit positions. Verified analytically:
  `((hi ^ ws_hi) << 4) | (lo ^ ws_lo) = byte ^ ((ws_hi << 4) | ws_lo)` ✓
