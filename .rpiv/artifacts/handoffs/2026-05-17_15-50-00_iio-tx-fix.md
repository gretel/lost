# Handoff: IIO TX fix — root cause found and fixed

## What was done

### TX loopback fix (COMPLETED)

**Root cause**: `tx_queue_depth = 0` in config → `TxRequestQueue(capacity=0)` always full → all TX requests rejected with "tx_queue_full" NACK. This is why no tx_ack was ever seen in the handoff testing.

**Secondary fix**: TX RF bandwidth in IIOSink was set to `cfg.rate` (sample rate, 2.5 MS/s) instead of `cfg.bw` (LoRa bandwidth, 62.5 kHz). Fixed to use `cfg.bw`.

**Changes made** (commit `9945f03`, 2 files):

1. `apps/config-iio-local.toml`:
   - `enable_tx = false` → `enable_tx = true`
   - `tx_queue_depth = 0` → `tx_queue_depth = 8`
   - Added `tx_lo_powerdown = false` (keep LO on between bursts)
   - `rx_gain = 60` → `rx_gain = 71`

2. `apps/lora_trx.cpp`:
   - IIOSink `bandwidth` property: `cfg.rate` → `cfg.bw`

### AA filter analysis (RESEARCHED)

Half-band filter cascade analysis confirmed:

- Current 23-tap ×6: -6 dB edge droop at 31.25 kHz (BW edge)
- Half-band constraint (even taps = 0) fundamentally limits passband to ~fp < 0.40 × Nyquist for short lengths
- For fp = 0.48: need ~180+ taps per stage → 1080 taps total
- Single-stage polyphase decimator would work better but needs different C++ implementation
- TX fix provides strong local signal → AA PMR loss becomes irrelevant

## Learned

1. **`tx_queue_depth = 0` prevents all TX** — TxRequestQueue(0) immediately full. Set to 8 for production, any positive value for testing.

2. **TX bandwidth should match LoRa BW** — IIOSink AD9361 filter bandwidth set to `cfg.bw` (62.5 kHz), not `cfg.rate` (2.5 MS/s). Reduces TX noise.

3. **Pluto FISH Ball TX/RX are isolated** — No loopback without physical cable connection between ports. Cannot verify AA decode quality remotely.

4. **rsync SSH key freshness** — SSH connections may fail with `Permission denied` after several rapid connections. Wait 1-2s between attempts or use `PreferredAuthentications=password`.

5. **CRITICAL: Local IIO buffer-boundary discontinuities** — `uri=local:` (local backend) has **buffer-boundary discontinuities** not present with `uri=ip:*` (network backend). Each `_buf.refill()` returns one kernel buffer via `read()` on the IIO char device. Boundaries between consecutive `read()` calls can drop or duplicate samples. Non-blocking mode (`non_blocking=true`) makes this worse via partial refills.

   Remote IIO is continuous because iiod on the device reads kernel buffers and **stitches them into a TCP stream** — the client sees a seamless byte stream with no boundaries.

   This is why the same OS=40 stride decoder fails on local IIO (data corruption → wrong bin values → checksum fail) but works on remote IIO. It also explains the handoff finding: "Same ADVERT frame (3× TX) → 3 different decoded headers on tezuka IIO."

   Fix options:
   - **Ring buffer in IIOSource**: accumulate across refills, emit contiguous blocks (most correct)
   - **Larger kernel buffers**: `buffer_size=262144` (2 MB) → refills every ~100ms vs ~13ms at 2.5 MS/s → fewer boundaries
   - **Blocking mode with short timeout**: `non_blocking=false, timeout_ms=100` — `refill()` blocks 100ms max, returns full buffer, watchdog config keeps scheduler alive
